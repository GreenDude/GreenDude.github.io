AI Agents vs. Purpose-Built Automation

Published on December 22, 2025

Let’s not use tomahawks when we can use a scalpel.

AI has rapidly found its way into delivery organizations. From meeting summaries to roadmap drafts, from backlog refinement to stakeholder communication, AI agents are increasingly proposed as a way to “automate” large parts of project and product management.

Automation itself is not new. As a former QA and Project Manager, I’ve relied on automation for years—long before AI became fashionable. The difference today is not whether we automate, but how we do it. This article is not anti-AI. It is anti-misuse. Because when it comes to delivery workflows, precision beats brute force.

Automation Is Not One Thing

There is a tendency to lump all automation together under the same umbrella. In practice, there are two very different approaches:

  1. Purpose-built, deterministic automation
    Scripts, integrations, templates, scheduled jobs, API-driven tools.
  2. AI-agent-driven automation
    Systems that interpret data, generate narratives, or act autonomously based on probabilistic reasoning.

They solve different problems—and confusing them creates risk.

What Purpose-Built Automation Does Well

Purpose-built automation is boring—and that’s a compliment; Pulling sprint data via an API, calculating metrics, filling a report template, and sending it on a schedule is:

  • deterministic
  • auditable
  • predictable
  • secure
  • cheap to run

A Practical Pattern: Using ChatGPT to Build “Scalpel Automation” (Even If You’re Not Technical)

You don’t need to be an engineer to benefit from purpose-built automation. If you can clearly describe what you want, ChatGPT can help you generate a simple Python script that can do many things. As an example, let's say we need a tool that pulls sprint metrics from Jira and renders a stakeholder-ready PDF report.

The key is to prompt like an operator: specify inputs, outputs, constraints, and what “done” looks like.

What to Ask ChatGPT For

Copy-paste this prompt (then replace the placeholders): 
You are a senior automation engineer. Write a Python 3 script that generates a Sprint Report PDF from Jira.

Environment:
- Python 3.11
- Use only common libraries: requests and reportlab (or fpdf2)
- Script runs locally from terminal

Inputs (via env vars):
- JIRA_BASE_URL (example: https://mycompany.atlassian.net)
- JIRA_EMAIL
- JIRA_API_TOKEN
- JIRA_BOARD_ID
- SPRINT_ID (or, if not provided, pick the last closed sprint)
- OUTPUT_PDF (default: sprint-report.pdf)

Metrics to extract:
1) Sprint name, start date, end date
2) Committed vs Completed issues (count and story points)
3) Spillover issues (not completed)
4) Added after sprint start (scope change)
5) Bugs vs Stories split
6) Top 10 completed issues list (key, summary, assignee)

PDF requirements:
- One-page executive summary + second page details
- Clear headings and tables
- Include a small “Notes” section placeholder at the end

Quality requirements:
- Robust error handling (401, 403, 404, rate limits)
- Validate missing env vars and print friendly messages
- Avoid printing secrets
- Include a README section at the top of the file explaining how to run it

Deliverable:
Provide the complete script as a single file, plus a sample .env file and example command to run it.

But if even this sounds complecated. Even this can be done with AI, just ask it to provide some guidance.

Where AI Agents Introduce Risk

1. Loss of Context
Project management depends on nuance—team dynamics, unspoken blockers, shifting priorities. AI can process text, but it cannot sense context the way humans do.
2. Non-Deterministic Output
AI systems are probabilistic. That variability is acceptable for drafting language, but dangerous for planning and reporting.
3. Silent Failure Modes
Scripts fail loudly. AI fails quietly—by omission, distortion, or hallucination.
4. Blurred Accountability
When an AI agent influences priorities or commitments, accountability becomes unclear. In delivery, that is a liability.

Where AI Agents Actually Make Sense

AI works best when it interprets, not when it operates.

  • Turning validated metrics into readable narratives
  • Drafting stakeholder updates
  • Summarizing meeting transcripts (with review)
  • Supporting backlog refinement
  • Identifying patterns in retrospectives

The Scalpel vs the Tomahawk

Using AI agents to replace deterministic automation is like using a tomahawk for surgery. Powerful—but reckless when precision tools either already exist or can be easily crafted. Purpose-built automation is the scalpel. AI agents must be carefully constrained to avoid collateral damage.

A Balanced Operational Model

Before we attempt to automate anything, we need to separate our responsibilities.

Automation
✅ data collection
✅ metrics and reporting
✅ scheduled processes
✅ compliance workflows
AI Agents
✅ summarization
✅ interpretation
✅ drafting
✅ pattern discovery
Humans
✅ judgment
✅ prioritization
✅ accountability
✅ team health

Final Thoughts

AI agents are a powerful complement—but not a replacement—for deterministic automation. Delivery systems fail not because teams lack intelligence, but because they introduce unnecessary complexity where simplicity already works.

Use AI deliberately. Use automation precisely.

Let’s not swing tomahawks when a scalpel does the job better.
